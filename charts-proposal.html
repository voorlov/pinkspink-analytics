3. Найденные нестыковки в текущем коде

Сравнил все 29 графиков. Вот четыре места, где сейчас разнобой — задача систематизации в первую очередь про них.

Issue 1
Форматтеры значений написаны по-разному для одинаковых типов данных
В коде нашёл минимум 11 разных форматтеров — для процентов, секунд, чисел. Часто отличаются только в мелочах: один с toFixed(1), другой без; один скрывает нули через v > 0, другой через v != null.
Сейчас (фрагменты разных графиков)
v => v
v => v > 0 ? v : ''
v => v + '%'
v => v.toFixed(1) + '%'
v => v != null ? v.toFixed(1) + '%' : ''
v => v > 0 ? v + '%' : ''
v => v > 0 ? v + 's' : ''
v => v != null ? v + 's' : ''
Предложение (4 единых функции)
fmtNum(v)      // 12 845
fmtPct(v, 1)   // 12.4%
fmtSec(v)      // 31s
fmtCur(v)      // $630

// Все скрывают нули и null
// автоматически
Issue 2
Скрытие нулей — где-то есть, где-то нет
Нули на графиках — визуальный шум. В коде часть графиков прячет их (v > 0 ? v : ''), часть — нет. У grouped bar воронки нули регулярно встречаются (этап «purchase» в неудачные недели = 0).
Предложение: правило по умолчанию — всегда скрывать нули и null. Реализуется внутри стандартных форматтеров (см. Issue 1).
Issue 3
Datalabel-позиции для bar — anchor/align не консистентны
Для grouped bar нашёл вариант anchor:'end', align:'top' (подпись над столбиком). Для stacked bar — anchor:'center', align:'center' с белым цветом (внутри сегмента). Это правильно и оба варианта нужны, но не зафиксировано как правило.
Предложение: правило-таблица — для каждого типа графика своя пара anchor/align, чтобы не выбирать каждый раз заново.
Issue 4
Цветовая логика — иногда из CHANNEL, иногда из FUNNEL, иногда хардкод
У большинства графиков цвета берутся из CHANNEL_COLORS или FUNNEL_STAGES (через TOKENS). Но в нескольких местах остался хардкод (например, borderColor: '#6C5CE7' вместо cssvar('--c-channel-social') или TOKENS["channel"]["social"]).
Предложение: правило — все цвета графиков берутся ИСКЛЮЧИТЕЛЬНО из CHANNEL_COLORS / FUNNEL_STAGES / TOKENS["semantic"]. Никаких inline hex.

4. Предложение конвенций

Что предлагаю зафиксировать в design-system/charts/ v1.0. Это не догма — обсуждаемо, можно править.

Convention 1
Стандартный набор форматтеров
Четыре функции, которые покрывают 90% случаев. Объявляются один раз в Chart.defaults или в общем JS-блоке, используются везде.
const fmt = {
  num: (v)         => (v == null || v === 0) ? '' : v.toLocaleString('ru-RU'),    // 12 845
  pct: (v, dec=1)  => (v == null || v === 0) ? '' : v.toFixed(dec) + '%',         // 12.4%
  sec: (v)         => (v == null || v === 0) ? '' : v + 's',                      // 31s
  cur: (v, c='$')  => (v == null || v === 0) ? '' : c + v.toLocaleString('ru-RU') // $630
};
Convention 2
Datalabel-позиции по типу графика
Правило-таблица. Один раз решили — везде применяем.
// Grouped Bar       → anchor:'end',    align:'top'    color:tx-primary
// Stacked Bar       → anchor:'center', align:'center' color:tx-ondark
// Combo (bar часть) → display:false (слишком плотно)
// Line              → align:'top'                     color:tx-primary
// Bubble            → display:false (используется label в tooltip)
// Sparkline         → display:false
Convention 3
Легенда — единое правило
Везде position:'top'. Исключение — sparkline (display:false).
Convention 4
Цвета — только через токены
Никаких inline hex в Chart-конфигах. Только cssvar('--c-channel-X') в JS, или TOKENS["channel"]["X"] при сборке в Python. Один источник правды.
Convention 5
Размеры (уже зафиксировано в типографике)
Datalabels = --fs-caption (11px). Оси и легенда = --fs-chart-meta (10px). Уже работает через Chart.defaults.
Convention 6
Tooltip — кастомный callback там, где есть % или валюта
По умолчанию tooltip Chart.js показывает голое число. Для процентов и валюты — custom callback с правильным форматированием. Можно тоже унифицировать через общую функцию.
tooltip: { callbacks: {
    label: ctx => ctx.dataset.label + ': ' + fmt.pct(ctx.parsed.y, 1)
}}

5. Вопросы для обсуждения

Шесть точек, по которым надо принять решение, чтобы зафиксировать в спецификацию. Прочитай и скажи: согласен / не согласен / предлагаю другое.

Q1
Принимаем 4 стандартных форматтера?
fmt.num · fmt.pct · fmt.sec · fmt.cur — все автоматически скрывают нули и null. Вместо 11+ разных формул будет 4 функции на весь дашборд. Локаль ru-RU для разделителя тысяч (12 845).
Q2
Скрываем нули — везде по умолчанию?
Если значение = 0 или null → подпись пустая. Это станет частью стандартных форматтеров. Альтернатива — показывать «—» вместо пустоты. Что выбираем?
Q3
Datalabel-позиции по типу — фиксируем таблицей?
См. Convention 2. Каждый тип получает раз и навсегда определённую пару anchor/align/color. Если ОК — применим ко всем 29 графикам в один проход.
Q4
Легенда всегда сверху?
Кроме sparkline — везде position:'top'. Альтернативы: bottom, right (вертикально). Top кажется оптимальным для дашбордовых графиков, но проверь — может, где-то bottom удобнее?
Q5
Tooltip — кастомизировать или дефолтный?
Дефолтный показывает голое число. Кастом с форматтером (+ '%', $, разделители тысяч) — приятнее, но больше кода. Пишем общую функцию customTooltip(fmt)?
Q6
Цветовая логика по типу датасета?
Правила: «датасет про каналы → CHANNEL», «про этапы воронки → FUNNEL», «про дельты → SEMANTIC (growth/decline)», «один тренд (sparkline) → SEMANTIC по направлению». Согласовываем?
