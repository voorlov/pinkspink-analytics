2. Восемь проблем

Issue 1
h2 имеет padding-bottom: 6px без border
В styleguide у h2 есть border-bottom: 2px solid — там 6px нужны как отступ от черты. В дашборде черты нет, а 6px остались — лишняя пустота между h2 и .meta.
Issue 2
h4 margin-bottom: 6px (хардкод)
Все остальные отступы в дашборде из шкалы 4/8/12/16/24/32. Только у h4 — 6. Должно быть либо --sp-1 4, либо --sp-2 8.
Issue 3
h3 без margin-top
Когда перед h3 идёт элемент со сброшенным margin (например, * { margin: 0 } на form-элементах) — h3 «прилипает» сверху. Сейчас работает только потому что предыдущие .meta даёт margin-bottom: 12.
Issue 4
Inline style="margin-top:32px" на двух h2
В report_week.html найдено 2 места: «Scroll Rate по сайту», «Cohort Retention». Дефолтных 24px не хватает после графиков с большой нижней областью. Сигнал: либо нужно везде 32, либо проблема в том что большие графики не дают своих margin'ов снизу.
Issue 5
.title-area h3 переопределяет шрифт
Глобальный h3 = 14 / semibold. В .title-area h3 = 13 / bold. Один тэг — два разных вида. Это нарушает иерархию: внешние block-headings 14px, внутренние 13px.
Issue 6
Двойственный .meta
Глобальный .meta = 13 / mb 12. В .title-area .meta = 11 / без mb. Один селектор — два смысла. Каскад работает, но именование не отражает разницу ролей.
Issue 7
Зазор после h2 больше чем после .meta
h2 → .meta = 12 + 6 (padding-bottom) = 18px. .meta → контент = 12px. Контр-интуитивно: «связанные блоки» (h2 и его описание) дальше, чем «разные блоки» (описание и контент). Должно быть наоборот.
Issue 8
h1 определён в CSS, но не используется в HTML
10 строк мёртвого кода. Можно удалить либо оставить как backup для возможного return.

3. Предлагаемые правила

Принцип: tight pair — заголовок и его описание считаем одним блоком, между ними --sp-1 (4px). Block break — между описанием и контентом --sp-3 (12px). Section break — между секциями --sp-6 (32px перед h2).

5. Принцип «tight pair»

Tight pair --sp-1 4px — заголовок и его описание/список считаются одним блоком, прижаты друг к другу. Применяется: h2 → .meta · h3 → .meta · h4 → .meta · h4 → list
Block break --sp-3 12px — между «section block» (h2 + .meta) и контентом (grid/table/chart). Применяется: .meta → контент
Sub-section break --sp-4 16px — перед h3 (когда используется как глобальный заголовок, не в карточке)
Section break --sp-6 32px — перед h2 (новая секция). Заменяет inline style="margin-top:32px" на правило в CSS.

6. Что добавится в Typography v1.0

Если согласуем — секция в tokens.md и tokens.json:

// === Headings & Spacing ===

h1 { /* удалён, не используется */ }

h2 {
    font-size:    var(--fs-h2);
    font-weight:  var(--fw-semibold);
    margin:       var(--sp-6) 0 var(--sp-1);  /* 32 / 4 */
}

h3 {
    font-size:    var(--fs-h3);
    font-weight:  var(--fw-semibold);
    margin:       var(--sp-4) 0 var(--sp-1);  /* 16 / 4 */
}

h4 {
    font-size:    var(--fs-body);
    font-weight:  var(--fw-semibold);
    margin:       var(--sp-2) 0 var(--sp-1);  /* 8 / 4 */
    color:        var(--tx-secondary);
}

.meta {
    font-size:    var(--fs-body);
    color:        var(--tx-secondary);
    margin-bottom: var(--sp-3);                /* 12 */
}

/* Tight pair — описание сразу после заголовка не дублирует gap */
h2 + .meta, h3 + .meta, h4 + .meta { margin-top: 0; }

/* Card title block — компактная вариация для .title-area внутри .cell */
.title-area {
    display: flex; flex-direction: column;
    padding-bottom: var(--sp-1);
}
.title-area h3 {
    font-size:    var(--fs-h3);                /* НЕ переопределяем — = глобальному */
    font-weight:  var(--fw-semibold);
    margin: 0 0 2px 0;
}
.title-area .meta {
    font-size:    var(--fs-caption);            /* меньше — это «подпись» */
    margin: 0;
    line-height: 1.3;
}

7.6 Откуда такой большой gap — в одной фразе

Grid-system в дашборде заставляет .title-area занимать 80px row, а justify-content: flex-end прижимает контент к низу. Эти 80px — не из spacing-токенов, а из дизайн-решения «все блоки в сетке имеют одинаковую высоту строки 80px» (чтобы графики и таблицы выравнивались).

Проблема: для табличных подсекций (как этот пример) 80px title-area — overhead. Контент h3+meta весит ~37px, остальные 43px — пустота. Для чартов это работает (визуальный ритм), но для таблиц где cell сам растёт по контенту — лишний воздух.

7.7 Что предлагаю в спецификации

ОпцияЧто меняемЭффект
A · Status quo Ничего не трогаем Воздух в title-area остаётся. Простое решение если визуально устраивает.
B · Top-align .title-area justify-content: flex-start (вместо flex-end) h3 прилипает СВЕРХУ. Воздух перемещается ВНИЗ (между meta и cell). Чисто визуально tight pair восстанавливается, но gap до cell растёт.

.meta-sm {
    font-size:    var(--fs-caption);    /* 11px */
    color:        var(--tx-secondary);
    line-height:  1.3;

8. Твоя структура отступов · em-based (новое предложение)

Три твои правки: (1) текст после h2 и h3 одинакового размера = 11px caption везде; (2) line-height = 1.25; (3) margin-top заголовков считается от их font-size (h2 = 200%, h3 = 150%). Это меняет спецификацию: уходит разница между .meta и .meta-sm (они сливаются в один .meta на 11px), а отступы заголовков становятся пропорциональными размеру шрифта через em.

8.1 Структура (твоя схема)

Правило округления: процент × font-size, затем округляем до ближайшего чётного числа в меньшую сторону. Так получаются «дизайнерские» значения которые легко складываются с пиксельной шкалой.

[предыдущий блок]
↕ 250% × h2 = 250% × 18 = 45 → 44px (округление вниз до чётного) — section break
<h2>
↕ 4px — tight pair
<p class="meta"> (11px / line-height 1.25)
↕ 175% × h3 = 175% × 14 = 24.5 → 24px (округление вниз до чётного) — sub-section break
<h3>
↕ 4px — tight pair
<p class="meta">
↕ 8px — content gap
[контент: table / grid / chart]

.meta margin-bottom 12px (--sp-3) 8px Меньше зазор до контента. С 11px шрифтом 8px достаточно.

8.5 Сводка CSS

Правило margin-top заголовков: процент × font-size, округление вниз до ближайшего чётного. Получаются «дизайнерские» значения (44 / 24 / 16) вместо прямого em.

body { line-height: 1.25; }              /* применяется ко всему */

h2 {
    font-size:   var(--fs-h2);            /* 18 */
    font-weight: var(--fw-semibold);
    margin:      44px 0 0;                 /* 250% × 18 = 45 → 44 (round even down) */
    line-height: 1.25;
}

h3 {
    font-size:   var(--fs-h3);            /* 14 */
    font-weight: var(--fw-semibold);
    margin:      24px 0 0;                 /* 175% × 14 = 24.5 → 24 */
    line-height: 1.25;
}

h4 {
    font-size:   var(--fs-body);          /* 13 */
    font-weight: var(--fw-semibold);
    margin:      16px 0 0;                 /* 125% × 13 = 16.25 → 16 */
    color:       var(--tx-secondary);
    line-height: 1.25;
}

.meta {
    font-size:    var(--fs-caption);      /* 11px ВЕЗДЕ — единый размер */
    color:        var(--tx-secondary);
    line-height:  1.25;
    margin:       0 0 8px;                 /* 0 top, 8px bottom (gap к контенту) */
}

/* Tight pair — описание сразу после заголовка */
h2 + .meta, h3 + .meta, h4 + .meta { margin-top: 4px; }

В виде токенов: можно вынести в --mt-h2: 44px; --mt-h3: 24px; --mt-h4: 16px; — тогда если меняешь размер заголовка, нужно вручную пересчитать токен по формуле floor_even(percent × size).

8.6 Для grid system (.title-area внутри .h4x6) — отдельно

Эти правила работают вне grid system (для разделов из секции 7.5 — flow layout). Внутри grid system .title-area с её 80px row продолжает существовать как отдельный паттерн (или решаем что с ним делать по вариантам B/C/D из секции 7.7).

Для согласования: применяем эту структуру (8.1-8.5) ко всем заголовкам ВНЕ grid system. Внутри grid (.title-area) — отдельный вопрос: либо унифицируем (поменяем .title-area .meta на 11/lh 1.25 — что и так), либо переделываем grid (варианты B/C/D из 7.7). Какой выбираем?

9. Вопросы для решения

Q1
Унифицировать h3 в карточке (14/semibold) или оставить компактный (13/bold)?
Унифицировать — иерархия чище, один h3 везде. Оставить — карточки плотнее, заголовок не «доминирует» в маленькой области. Я предлагаю первое (см. Pattern B). Но если 14px реально съест воздух в карточке — оставим 13/bold и переименуем правило (например, .title-area h3 = --fs-cardhead).
Q2
«Tight pair» = 4px (--sp-1)?
Альтернативы: 0 (приклеить вплотную), 6px (мягче), 8px (--sp-2). 4px — стандарт «связанная пара» в большинстве дизайн-систем (Material, Polaris). Принимаем?
Q3
Section break перед h2 = 32px?
Сейчас в коде 24, в HTML inline 32 в двух местах. Если ставим 32 везде — те inline можно убрать, заодно решается «странная пустота» в этих двух блоках. Принимаем 32?
Q4
Удалить h1 из CSS?
Не используется в дашборде. Удалить — чище. Оставить — backup на будущее. Я бы удалил.
Q5
Принимаем .meta-sm как новый класс?
11px caption variant of .meta — для коротких tag-like подписей (как «∆ vs avg прошлых 4 недель» в скриншоте). Альтернатива — оставить только .meta 13px, мириться что в плотных подсетках текст крупнее. .meta-sm даёт визуальную плотность без ухода в overrides.
