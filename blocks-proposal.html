2. Найденные нестыковки

Issue 1
Три класса для одного блока
.h4x6 (placement) + .title-area (header layout) + .cell (visual) — три обёртки для одного семантического блока «карточка с графиком». Сложно поддерживать, легко забыть один класс.
Issue 2
80px row хардкод создаёт пустоту
.grid { grid-auto-rows: 80px } + .h4x6 { grid-template-rows: 80px 1fr } + .title-area { justify-content: flex-end }. Контент h3+meta весит ~37px, но row фиксирован 80px → ~43px пустоты сверху в каждом блоке. Это «странные большие отступы» в скриншоте.
Issue 3
.title-area переопределяет шрифты — нарушает Typography v1.1
.title-area h3 { font-size: var(--fs-body); font-weight: bold } — 13/bold вместо global 14/semibold. .title-area .meta { line-height: 1.3 } — не 1.25 как у global .meta. Заголовок внутри блока выглядит мельче и жирнее чем заголовок в обычном flow, хотя должен быть такой же.
Issue 4
Grid-row span форсит высоту блока
.h4x6 { grid-row: span 5 } = 5 × 80 = 400px независимо от контента. Если внутри график 200px — пустота 200px снизу. Если контент по природе короче — обёртка всё равно растянется.
Issue 5
Inline overrides на h2 (workaround)
В двух местах <h2 style="margin-top:32px;"> — потому что дефолтный margin-top h2 не подходил после chart-блоков. С новой типографикой --mt-h2 = 44px, эти inline overrides побеждают токен и оставляют 32px. После миграции — убрать.

3. Предложение конвенций · Blocks v1.0

Один примитив .block взамен .h4x6 + .title-area + .cell. .grid теряет хардкод grid-auto-rows: 80px — ряды auto по контенту. Высота контента задаётся через chart-tokens (--ch-md и т.п.) — визуальный ритм сохраняется через единые размеры графиков, а не через grid-row span.

ПараметрСейчасСтало

Grid auto-rows 80px (хардкод) убран — auto по контенту
Заголовок внутри .title-area h3 (13/bold override) обычный h3 + правила Typography v1.1
Описание .title-area .meta (lh 1.3) обычный .meta + Typography v1.1 (lh 1.25)
Высота grid-row span 5 = 400px фикс по контенту (chart-wrap задаёт высоту через --ch-md)
Padding блока 14px (хардкод) --sp-3 12px (из шкалы)
Inline overrides на h2 2 места удаляются — токен --mt-h2: 44px работает

5. Совместимость с другими разделами

РазделСовместимостьКомментарий
Typography v1.1 ✓ полная Внутри .block работают глобальные h3, h4, .meta. .block > :first-child { margin-top: 0 } — техническое правило чтобы первый h3 не получал свой 24px margin-top (он внутри блока).

Controls v1.0 ✓ полная Контролы (filter-bar, кнопки) живут вне .block, в нормальном flow. Никакого пересечения.

7. Вопросы для решения

Q1
Принимаем единый .block?
Заменяет цепочку .h4x6 + .title-area + .cell одним классом. Padding 12px, radius 8px, shadow card. Внутри — h3 + meta + контент в normal flow.
Q2
Убираем grid-auto-rows: 80px?
Ряды grid'а становятся auto по контенту. Визуальный ритм сохраняется через единые chart-heights (--ch-md для большинства, --ch-lg для bubble и т.п.).
Q3
Span-классы (.span-12 / .span-8 / .span-6 / .span-4)?
Заменяют .h4x12 / .h4x8 / .h4x6 / .h4x4. Имена короче и универсальнее (.span-N = grid-column span N). Высота больше не закодирована в имени класса — задаётся отдельно через .h-md и т.п.
Q4
Padding блока — --sp-3 12px?
Сейчас хардкод 14px. --sp-3 12px — из spacing-шкалы. Альтернатива — --sp-4 16px (просторнее). 12px ближе к текущему 14px, переход незаметный.
Q5
KPI блоки (h2x2, h1x4, h1x2) — отдельным спеком?
У них специфическая внутренняя раскладка (kpi-label + kpi-value + kpi-spark). Сейчас в дашборде не используются на main view. Предлагаю: .block — основной примитив для chart/table блоков. KPI-варианты (со спарклайнами и т.п.) — будущий раздел components/.
Q6
Migration: всё одной волной или поэтапно?
26 .h4x6 + 5 .h4x12 + 1 .h4x8 + 37 .title-area = много замен. Можно: (а) одной волной с глобальным sed по generate_report.py, (б) по группам (сначала h4x6, потом h4x12 и т.д.). Какой путь?
